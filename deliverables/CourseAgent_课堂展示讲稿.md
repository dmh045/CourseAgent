# CourseAgent 课堂展示讲稿

各位老师、同学大家好，我今天展示的项目叫 CourseAgent，它是一个面向课程资料和课堂录屏的智能学习材料生成工作台。

这个项目最初想解决的问题很直接：在学习软件工程、交互设计这类课程时，我们经常会拿到 PDF、PPT、课堂录屏、字幕等资料，但这些资料本身比较分散。学生如果要复习，需要自己读文档、提炼重点、整理思维导图、做汇报 PPT，甚至还要回看很长的视频去找关键时间点。CourseAgent 的目标就是把这些重复性的整理工作变成一个可视化的一键流程：用户上传资料，系统自动解析内容，并生成摘要、关键词、思维导图、答辩 PPT、讲稿、讲解视频、字幕、重点时间戳和历史记录。

我会从三个方面介绍这个项目：第一，系统能做什么；第二，背后的技术架构是怎么设计的；第三，项目在实现过程中遇到的问题，以及我是怎么改进的。

## 一、项目功能演示思路

首先进入系统页面，可以看到它不是一个单纯的文件转换工具，而是一个工作台式的界面。页面分成四个主要区域：课程文档生成、课堂视频分析、产物中心和运行详情。

在课程文档生成部分，用户可以上传 PDF、DOCX 或 TXT 课程资料，然后输入任务目标。比如我可以要求系统把这份资料整理成适合课堂展示的 PPT，同时生成摘要、关键词、思维导图、逐页讲稿和讲解视频。点击运行后，系统会自动完成文档读取、任务规划、内容生成、PPT 制作、视频合成和产物校验。

这里有一个比较重要的设计点：我没有把每个功能做成彼此独立的按钮，而是把它组织成一个 Agent 工作流。也就是说，用户只需要描述最终目标，系统会根据目标判断需要调用哪些工具。例如，如果任务目标里提到了 PPT，系统会生成 PPT 大纲和 PPT 文件；如果提到了视频讲解，系统会继续生成讲稿、字幕和讲解视频；如果需要知识结构，系统会生成思维导图。

第二个部分是课堂视频分析。这个模块面向的是课堂录屏或教学视频。用户可以上传视频，也可以上传 SRT 字幕。如果没有字幕，系统会尝试使用本地 Whisper 语音识别生成字幕。得到字幕后，系统会分析课堂内容，提炼课程总结、核心知识点、重点时间戳、Highlight 复习片段和视频思维导图。这样用户复习时就不必从头拖完整个视频，而是可以直接根据时间戳定位重点内容。

第三个部分是产物中心。每次分析都会生成一个独立的 run_id，并把该次运行的结果记录到 manifest 文件中。这样做的好处是，课堂展示前可以提前生成所有产物，不用担心现场重新分析耗时过长；同时，文档分析和视频分析的产物不会互相覆盖，因为每次运行都有独立目录和独立索引。

第四个部分是运行详情。这里展示的是 Agent 决策、执行轨迹、摘要内容、PPT 大纲、执行日志和全部下载入口。这个区域不是为了普通用户“看热闹”，而是为了让系统的智能过程可解释。用户可以看到系统为什么选择这些工具、每一步执行了什么、最后是否通过了产物完整性校验。

## 二、系统总体架构

从架构上看，CourseAgent 可以分成五层：交互层、Agent 编排层、AI 推理层、工具执行层和数据持久化层。

交互层使用 Streamlit 实现。选择 Streamlit 的原因是它适合快速构建数据应用和 AI 工具工作台，文件上传、按钮交互、进度反馈、下载按钮和状态展示都可以较快完成。对这个项目来说，Streamlit 的价值不只是“做页面”，而是把文档处理、视频分析和历史记录组织成一个可以演示、可以操作、可以追溯的完整流程。

Agent 编排层是项目的核心。它不是简单地把函数顺序写死，而是采用 Planner、Tool Executor、Verifier、Auto-Repair 这样的流程。Planner 负责根据用户目标和文档内容决定工具链；Tool Executor 按计划执行文档解析、摘要、关键词、导图、PPT、视频等工具；Verifier 会检查计划中的产物是否真实生成、文件是否存在、是否为空；如果发现缺失，Auto-Repair 会尝试重新执行对应工具。这个设计让系统从“模块集合”更接近一个可解释的 Agent。

AI 推理层使用兼容 OpenAI 接口的 LLM 调用方式，同时支持 OpenAI、Gemini、DeepSeek 和自定义 OpenAI-compatible API。这样做的好处是模型供应商可以切换，不会被单一 API 绑定。项目里还设计了本地 fallback：当没有 API Key、API 报错或额度不足时，系统不会完全停摆，而是使用本地规则生成基础摘要、关键词和结构化内容。这样在课堂演示时，即使 API 不稳定，也能保持基本演示能力。

工具执行层负责把 AI 生成的结构化内容变成真实文件。文档读取支持 PDF、DOCX 和 TXT；PPT 生成使用 python-pptx，把大纲、重点和导图写入演示文稿；思维导图优先使用 Graphviz 渲染为图片，如果环境不支持，也可以退化为 Mermaid 文本；视频讲解部分使用 MoviePy 合成画面、音频、字幕和时间戳；课堂视频分析部分支持 SRT 字幕解析，也支持 faster-whisper 或 PyTorch Whisper 的本地语音识别。

数据持久化层采用 run_id 和 manifest 的方式管理产物。每次运行都会创建独立目录，里面保存原始输入、生成产物和 manifest.json。manifest 记录本次任务的类型、状态、源文件、产物路径、耗时、错误、Agent 决策和执行轨迹。这样 UI 可以通过 run_id 查询历史记录，用户也能随时回到某一次分析结果进行下载和展示。这个设计后续如果上线，也可以平滑迁移到对象存储和数据库：大文件放对象存储，manifest 和任务状态放数据库，后台任务用队列异步执行。

## 三、核心技术实现

这个项目的核心不是单一算法，而是多个技术点的组合。

第一是文档解析和结构化生成。用户上传资料后，系统先把文件统一解析为文本，再把文本交给 LLM 或本地规则进行处理。为了避免早期出现“生成内容和原文不匹配”的问题，我在提示词和校验逻辑里强调了两个原则：一是必须基于原文，不允许加入无关项目内容；二是输出要尽量结构化，使用 JSON 作为中间格式。这样后续生成 PPT、导图和讲稿时，不是从一段散文里硬拆内容，而是读取稳定的字段。

第二是 Agent 任务规划。Planner 会根据用户目标识别输出需求，例如摘要、关键词、思维导图、PPT、视频、字幕等，并生成 selected_tools。每个工具执行后都会写入 trace，包括 step、thought、action 和 observation。这样运行详情页面可以展示“系统为什么这么做”。这比单纯写一个 if-else 流程更适合课堂展示，因为它体现了 Agent 的决策过程。

第三是产物生成。PPT 不是简单把文字塞进幻灯片，而是先生成大纲，再根据页面主题、要点和讲稿进行排版。视频讲解也不是照着 PPT 生硬朗读，而是基于提炼内容生成讲稿，再合成讲解视频、字幕和时间戳。课堂视频分析则反过来：从视频或字幕提取文本，再生成知识点、时间戳和复习片段。

第四是质量校验和自动修复。早期系统经常出现只生成摘要、PPT 为空、思维导图过于简单、或者页面详情与产物不匹配的问题。后来我加入了 Verifier 检查机制：系统会检查计划中应该有的文件是否存在，例如 summary.md、keywords.json、mindmap.png、generated_presentation.pptx、final_video.mp4、subtitles.srt、video_markers.json 等。如果缺失，就记录错误并尝试自动修复。最近我还统一了文档生成、视频分析和历史记录的状态结构，使运行详情能正确显示 Agent 决策和执行轨迹。

第五是缓存和历史记录。文档生成通常比较耗时，尤其是调用大模型和生成视频时。为了减少重复等待，系统会根据源文件、用户目标、模型配置计算 fingerprint。如果相同任务已经成功运行过，就可以复用历史产物。产物中心则按 run_id 展示历史记录，避免文档产物和视频产物互相覆盖，也方便课堂展示前提前准备。

## 四、提示词设计与质量优化

这个项目中，提示词设计是非常关键的一部分。最开始生成效果并不好，例如文档是“交互设计过程”，但生成内容里却出现了 Agent 工作流程；有的 PPT 页内容过于空泛；思维导图也只是几个泛泛的关键词。

针对这些问题，我做了几类改进。

第一，提示词必须绑定原文。系统要求模型优先引用上传资料里的概念、章节和例子，不能把项目自身的技术词混进课程内容。这样可以避免“课程资料被系统介绍污染”的问题。

第二，输出结构必须清晰。摘要、关键词、PPT 大纲、讲稿、视频分析都尽量要求 JSON 格式输出，并规定字段含义。例如 PPT 大纲要包含页码、标题、要点和讲解意图；视频分析要包含 summary、key_points、markers 和 highlights。这样后续工具可以稳定消费这些结果。

第三，提示词要考虑教学场景。比如讲稿不能只是逐字朗读 PPT，而是要用更通俗的语言解释概念；PPT 不是堆文字，而是要围绕“课堂展示”组织叙事；视频时间戳不是随便分段，而是要标记值得复习的知识点和关键片段。

第四，增加本地质量兜底。即使 LLM 输出不稳定，系统也会检查思维导图是否过于稀疏、是否出现与原文无关的项目术语。如果发现问题，就回退到本地提取式逻辑，保证至少得到一份和原文相关的结果。

## 五、演示时可以重点展示的亮点

如果现场演示时间有限，我建议重点展示三个亮点。

第一个亮点是“一键生成完整学习材料”。上传一份课程 PDF 后，展示系统自动生成摘要、关键词、思维导图、PPT 和讲稿。这个部分最能体现项目的实用性。

第二个亮点是“Agent 可解释”。切换到运行详情页面，展示 Agent 决策和执行轨迹。可以说明系统不是黑箱地生成一个结果，而是记录了工具选择、执行过程和校验结果。

第三个亮点是“课堂视频分析”。上传视频或 SRT 后，展示课程总结、重点时间戳、Highlight 片段和视频思维导图。这个部分能说明项目不仅处理静态文档，也能处理课堂录屏这类多媒体学习资料。

## 六、项目问题与改进方向

目前项目还有一些可以继续完善的地方。

第一，生成质量仍然依赖模型能力和提示词设计。虽然系统已经加入了原文一致性校验和 fallback，但如果输入资料质量差、OCR 文本混乱，生成效果仍会受到影响。后续可以加入更强的文档结构识别，例如按标题、页码、图表和公式进行分块。

第二，视频分析的本地语音识别对机器性能有要求。Whisper 模型会占用较多磁盘和内存，课堂演示时更推荐提前准备 SRT 字幕。后续上线后，可以把语音识别放到异步后台任务或云端服务中，避免阻塞前端页面。

第三，当前数据存储主要基于本地目录和 manifest 文件，适合课程项目和本地演示。如果要上线给多用户使用，需要引入数据库、对象存储、任务队列和权限管理。大文件可以放对象存储，任务状态和产物索引放数据库，生成任务交给后台 worker 执行。

第四，UI 还可以继续增强。例如可以加入更细的进度条、任务队列状态、失败重试按钮，以及对 PPT 和视频结果的在线预览。这样用户不需要下载文件，也能快速判断结果是否可用。

## 七、总结

总的来说，CourseAgent 的定位是一个面向课程学习场景的智能内容生产工作台。它把文档解析、LLM 结构化生成、PPT 制作、视频合成、字幕分析、知识点提炼、历史记录和产物校验整合到同一个流程中。

我认为这个项目最有价值的地方不只是“能生成文件”，而是把学习资料处理变成了一个可追溯、可复用、可解释的 Agent 工作流。用户上传资料后，系统会规划任务、调用工具、生成产物、校验结果，并把每次运行沉淀为历史记录。这样既能提升学习资料整理效率，也能为后续扩展成在线课程助教系统打下基础。

我的展示到这里结束，谢谢大家。
